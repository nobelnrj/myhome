<objective>
Wire classification into the two rows that draw icon tiles, without ever letting inference block a draw.

`KitchenLogic.icon(forName:)` is a synchronous pure function called during row rendering; inference is async. AI-SPEC §4.4 calls this "the real work" of the phase. The resolver bridges the two: a synchronous cache-or-keyword answer that is always available immediately, plus a background classification that upgrades the tile in place when it lands.

Purpose: deliver the visible outcome of ICON-01 while structurally holding the ICON-03 non-blocking guarantee.
Output: `PantryIconResolver.swift`, both rows reading through it, and a test suite proving the list draws before classification completes.
</objective>

<locked_decisions>
Resolved from AI-SPEC §6, binding:

- **P22-D4 (lazy, not eager):** classification is triggered LAZILY on first render of a row, not eagerly on item creation. Eager classification would miss every row that arrives by sync import or by SchemaV11 data already on the device, and would put inference on the item-save path. Lazy triggering also makes the non-blocking guarantee the default rather than something to engineer.
- **P22-D6 (shared resolver instance):** one `PantryIconResolver.shared` serves both rows, read directly in `body` rather than plumbed through `@Environment`. `@Observable` tracks any read that happens during body evaluation, so no environment injection is needed, and a shared instance means a name appearing in both the pantry and shopping lists is classified once. Tests construct their own instance with injected dependencies.
</locked_decisions>

Create `MyHomeApp/Features/Kitchen/PantryIconResolver.swift`.

Declare `@MainActor @Observable final class PantryIconResolver`. Dependencies are injected for testability and defaulted for production: `init(classifier: PantryIconClassifying?, cache: PantryIconCache = PantryIconCache())`. A nil classifier means "model unavailable" — that is how the tests exercise the degraded path without needing a device.

Add `static let shared: PantryIconResolver`, whose classifier is built by checking availability once: if `#available(iOS 26, *)` AND `isPantryIconClassificationAvailable(SystemLanguageModel.default.availability)` is true, use `FoundationModelsPantryIconClassifier()`; otherwise nil. Do the availability check ONCE at construction — not per row, not per render.

State: a `private var resolved: [String: PantryCategory]` in-memory mirror seeded lazily from the cache (this is the observed property that drives the in-place upgrade), and a `private var inFlight: Set<String>` for coalescing. Both keyed by `KitchenLogic.normalizedIconKey`.

`func presentation(forName rawName: String?) -> (symbol: String, color: Color)` — SYNCHRONOUS, no `async`, no `await`, and it must never call the classifier. Order per AI-SPEC §4.2: normalised key → in-memory `resolved` → `cache.category(forName:)` (memoised into `resolved`) → `KitchenLogic.keywordCategory(forName:)` → `PantryCategory.other`. Map the winning category through `PantryCategory.presentation`. Add a `presentation(for item: PantryItem)` convenience so the row call sites stay one line.

`func classifyIfNeeded(name rawName: String?) async` — non-throwing by design so no call site needs a `try`. Returns immediately when: the name normalises to nil; the classifier is nil; the key is already in `resolved` or in the cache; or the key is already `inFlight`. Otherwise insert into `inFlight`, `defer` its removal, `await classifier.classify(name:)` inside a `do/catch` that swallows every error — a failed classification means the keyword tile simply stays, per AI-SPEC §4.2. On success write to `cache` and to `resolved` (the `@Observable` mutation is what re-renders the row). Check `Task.isCancelled` after the await and return before writing if cancelled.

Because the class is `@MainActor` and `resolved`/`inFlight` are only touched on the main actor, the coalescing set needs no lock.

Create `MyHomeTests/PantryIconResolverTests.swift`, reusing `FakePantryIconClassifier` from 22-02. Every test constructs its own resolver over an isolated `UserDefaults` suite — never `PantryIconResolver.shared`, which would touch the real app group and leak between tests. Mark the suite `@MainActor`. Cover every behaviour listed above; the coalescing test uses a `withTaskGroup` of ten concurrent calls, and the fake must `await Task.yield()` inside `classify` so the concurrency is real rather than serialised.

The non-blocking assertion AI-SPEC §5.2 asks for is the first behaviour in the list: call `presentation` and assert the fake's `callCount == 0` — proving the render path cannot reach the model at all, which is a stronger guarantee than timing a draw.

pbxproj: four manual edits per new file (PBXBuildFile, PBXFileReference, PBXGroup child, PBXSourcesBuildPhase); no synchronized groups exist.

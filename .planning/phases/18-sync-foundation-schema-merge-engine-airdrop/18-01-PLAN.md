SchemaV9 that appends `syncID: UUID` + `updatedAt: Date` to all 11 models and
introduces the `DeletionLog` tombstone @Model (SYNC-01); wire the V9→V10 migration

(`syncID`/`updatedAt` + `touch()`) that Plans 02–05 and Phases 19/20 build on.

entire merge engine (Plan 03) is keyed on these two fields existing and being
per-row distinct.

flipped, DeletionLog queryable.
</objective>

    preserving the FB13812722 workaround comment pattern.

    The didMigrate closure is REQUIRED (not nil) because of the SwiftData
    constant-default footgun: a `UUID()`/`Date()` default expression added via
    additive migration is evaluated ONCE, so every migrated row would share the SAME
    syncID/updatedAt — which breaks the entire sync identity model. didMigrate is
    SYNCHRONOUS (throws, NOT async). For each of the 11 SchemaV10 model types:
    1. Fetch all rows (`FetchDescriptor<SchemaV10.X>()` — destination types, as v5ToV6 does).
    2. syncID backfill, idempotent: track a `Set<UUID>` of seen syncIDs per table;
       first occurrence keeps its value, any row whose syncID is already seen gets
       `syncID = UUID()`. (First run: all rows share the constant default → all but
       one reassigned. Re-run after a mid-stage throw: all distinct → no-op.)
    3. updatedAt backfill, idempotent by construction: Expense — DO NOT touch (it
       carried a real updatedAt since V4). Note — `updatedAt = modifiedAt`.
       NoteBlock — `updatedAt = note?.modifiedAt ?? Date()`. All others —
       `updatedAt = createdAt`.
    4. Finish with explicit `try context.save()` (migration context does NOT
       auto-commit — v5ToV6 Pitfall 3).

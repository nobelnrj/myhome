<objective>
Ship the shopping half of the kitchen (KTCH-03): a Shopping segment inside
KitchenView that auto-populates from low/out-of-stock pantry items, restocks
the pantry when a derived row is checked off while shopping, and supports
manually-added extras backed by the synced `ShoppingListItem` model.

Honors the 20-01 locked design: the auto list is DERIVED + MANUAL. Low/out rows
are computed from PantryItem state at render time and are never written as
ShoppingListItem rows — so two phones can never create duplicate "auto" rows
for the merge engine to reconcile, and check-off restock is unambiguous because
a derived row IS its PantryItem. `ShoppingListItem` rows exist ONLY for manual
extras (which sync via 20-02's DTOs). Consequence, stated explicitly: checking
off a MANUAL extra never mutates the pantry — only derived rows restock
(the clear pantry↔shopping link the phase demands).

Check-off semantics for derived rows: quantity += restockQuantity
(KitchenLogic.markRestocked — one restock = one purchase, per the 20-01 field
design). If the item is still at/below threshold afterwards (restockQuantity
too small), it legitimately stays on the list — the user can tap again or fix
restockQuantity in the editor; this behavior is test-pinned, not accidental.

Purpose: completes the phase's user story — see what's low, shop from the
list, check off, pantry is current again — with all state changes carrying
honest sync clocks.

Output: segmented Kitchen host, shopping segment with derived + manual
sections, behavior tests GREEN, both-theme screenshots.
</objective>

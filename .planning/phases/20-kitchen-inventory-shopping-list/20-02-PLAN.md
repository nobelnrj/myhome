`SyncSnapshot` with `PantryItemDTO`/`ShoppingListItemDTO` arrays, add
`SyncEntityKind.pantryItem`/`.shoppingListItem`, bump
`SyncSnapshot.currentSchemaVersion` from 10 to 11 (lockstep with SchemaV11 from
20-01), and teach `SnapshotExporter`/`SnapshotImporter` to export, tombstone,
adopt, and LWW-upsert kitchen rows. Because the version stamp changes, snapshots
from a phone still on v10 are refused by the EXISTING version probe (18-02) —
no new refusal code needed, but the refusal must be re-proven at found: 10 /
expected: 11, and the 18-02/18-03 tests that hard-assert `expected: 10` must be
updated.

Identity adoption (mirrors 18-03's Category rule): both phones will independently
add the same staples ("Rice", "Milk") before their first sync. A remote kitchen
DTO with no local syncID match adopts a local row whose NAME matches after
trimming whitespace and case-folding; the survivor takes
`min(local.syncID.uuidString, remote.syncID.uuidString)` so both phones converge
to the same identity — the same vantage-independent min rule as Category/Expense
adoption.

Quantities stay `Double` end-to-end (20-01 design decision: pantry amounts are
not money; the SyncDecimal string rule applies ONLY to Decimal money fields).
DTOs pass Double through unchanged.

Purpose: this is the plan that makes KTCH-04 true — kitchen data flows through
the same tested engine as financial data, with the same no-duplicate,
no-resurrection guarantees.

Output: snapshot document layer at version 11 with kitchen DTOs; engine handles
kitchen end-to-end; new + updated sync tests GREEN.

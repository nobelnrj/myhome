## Summary

Phase 11.1 layers two services on the complete Phase 11 Asset Tracker: (1) a daily best-effort NPS NAV refresh sourced from npsnav.in, cloned from AMFINavService; and (2) a hybrid SIP accrual engine that auto-estimates unit purchases on elapsed installment dates, using exact historical NAV, and surfaces a monthly reconcile reminder through the existing NotificationScheduler.

The external API research is the highest-value finding here. Both npsnav.in and mfapi.in are verified live: npsnav.in exposes `https://npsnav.in/api/{code}` (plain text NAV), `https://npsnav.in/api/latest-min` (JSON array-of-pairs), `https://npsnav.in/api/historical/{code}` (JSON array of `{"date":"DD-MM-YYYY","nav":float}` in reverse-chronological order). mfapi.in exposes `https://api.mfapi.in/mf/{scheme_code}` returning `{"meta":{...},"data":[{"date":"DD-MM-YYYY","nav":"string"},...]}` in reverse-chronological order — no date-specific endpoint exists; client must binary-search the full history array. The AMFI historical endpoint (`portal.amfiindia.com/DownloadNAVHistoryReport_Po.aspx`) also works but returns the same semicolon format as NAVAll.txt, making mfapi.in the simpler choice.

The notification infrastructure is already complete. `NotificationScheduler` + `ReminderInfo` (both public structs in `MyHomeApp/Support/`) can schedule the monthly reconcile reminder with `.monthly` recurrence directly — no new notification code is needed. SchemaV8 follows the exact pattern of V7 (copy-all-verbatim, append new fields, add new @Models, `.custom` nil/nil migration stage, flip all typealiases atomically).

**Primary recommendation:** Clone AMFINavService → NPSNavService (swap URL + parser). Build SIPAccrualService as a new `@MainActor @Observable final class` parallel to RoutineResetService. Use mfapi.in full-history fetch + client-side binary search for MF date lookup (one network call per fund, cached per session). Wire both new services into the existing `RootView.scenePhase .active` chain. The monthly reconcile reminder schedules itself in SIPAccrualService using the existing `NotificationScheduler`.

### Pattern 1: NPSNavService — Clone of AMFINavService

**What:** `@MainActor @Observable final class` with `refreshIfNeeded()` / `forceRefresh()` entry points, IST daily gate via `UserDefaults`, `Task {}` wrapping URLSession, silent-fail.

**Source URL:** `https://npsnav.in/api/latest-min`

**Parser difference from AMFINavService:** npsnav.in `/api/latest-min` returns a JSON array-of-2-element arrays `[[code, nav], ...]` where `nav` is a raw number (not a string). Parse with `JSONDecoder` into `[[Any]]` or a custom `Decodable`.

**When to use:** Daily IST-gated refresh on scenePhase .active; forceRefresh on pull-to-refresh.

```swift
// Source: verified live at https://npsnav.in/api/latest-min (2026-06-12)
// Response shape (first 3 entries from live fetch):
// [["SM001001", 49.5429], ["SM001002", 42.6012], ["SM001003", 54.4013], ...]
// nav value is JSON Number (not String) — use Double intermediate, then Decimal(Double) or
// parse as NSDecimalNumber(floatLiteral:) to preserve precision for NAV display purposes.
// IMPORTANT: Decimal(floatLiteral:) is lossy. Use Decimal(string: String(nav)) after
// formatting to 4dp, or NSDecimalNumber(value: nav).decimalValue.

private static let npsLatestURL = URL(string: "https://npsnav.in/api/latest-min")!

// Decode array-of-arrays: [[schemeCode: String, nav: Double]]
private struct NPSEntry: Decodable {
    let code: String
    let nav: Double  // intermediate; convert to Decimal immediately after decode
    init(from decoder: Decoder) throws {
        var container = try decoder.unkeyedContainer()
        code = try container.decode(String.self)
        nav  = try container.decode(Double.self)
    }
}
// Parse: let entries = try JSONDecoder().decode([NPSEntry].self, from: data)
// Convert: Decimal(string: String(format: "%.4f", entry.nav))  — safe string-based conversion
```

**Staleness badge:** Already handled by `StalenessView` + `Asset.navAsOfDate`. `Last Updated` field in npsnav.in `/api/latest-min` metadata (the root object also has a top-level `lastUpdated` key). However since the simple array format does not include per-entry dates, use the `Last Updated` from the metadata response or set `navAsOfDate` to today's IST start when parsing.

**Alternative for date per-entry:** use `/api/detailed/{code}` (returns `Last Updated` per scheme) but that is 3 separate calls for 3 NPS holdings. Better: use `/api/latest-min` for bulk daily update; the staleness date is the fetch date, consistent with how AMFINavService works (it sets `navAsOfDate = scheme.navDate` from the file, but the file itself publishes one date for the fetch batch).

### Pattern 2: Historical NAV Lookup — SIPAccrualService

**What:** Given an installment date (IST), return the NAV published on that date or the nearest prior published date. Used by accrual engine to price each installment.

**MF source (mfapi.in):**

```
GET https://api.mfapi.in/mf/{amfiSchemeCode}
```

Response (verified live):
```json
{
  "meta": { "fund_house": "Axis Mutual Fund", "scheme_code": 120503, ... },
  "data": [
    {"date": "11-06-2026", "nav": "102.88080"},
    {"date": "10-06-2026", "nav": "103.18730"},
    ...
  ],
  "status": "SUCCESS"
}
```

- `data` is reverse-chronological (newest first). [VERIFIED: live fetch 2026-06-12]
- `nav` field is a String (unlike npsnav.in which uses Number). Parse with `Decimal(string:)`.
- Date format: `DD-MM-YYYY` (note: NOT `DD-MMM-YYYY` as in NAVAll.txt — different format).
- No date-specific endpoint exists. Fetch full history once per fund per accrual session; binary-search or linear-scan the array for the target date.
- Cache the history array in memory for the duration of the accrual pass (not persisted to SwiftData — stale after one session is fine; re-fetch on next accrual).

**NPS source (npsnav.in):**

```
GET https://npsnav.in/api/historical/{npsSchemeCode}
```

Response (verified live for SM001001):
```json
[
  {"date": "10-06-2026", "nav": 49.5429},
  {"date": "09-06-2026", "nav": 49.5869},
  ...
]
```

- Reverse-chronological order. [VERIFIED: live fetch 2026-06-12]
- `date` format: `DD-MM-YYYY`. `nav` is a Number (not String).
- Covers weekdays only (weekends/holidays absent — confirmed by gap from 05-Jun to 08-Jun across the weekend).

**Nearest-prior-date fallback algorithm:**

```swift
// Source: [ASSUMED] Foundation Calendar pattern, no external reference
// Both mfapi.in and npsnav.in history arrays are reverse-chronological.
// Find the first entry whose date <= targetDate (IST start-of-day).
func navEntry(for targetDate: Date, in history: [(date: Date, nav: Decimal)]) -> (date: Date, nav: Decimal)? {
    // history is sorted newest-first; first entry with date <= target is the nearest-prior
    return history.first { $0.date <= targetDate }
}
```

**Date parsing for history entries:**

```swift
// Date format for both mfapi.in and npsnav.in: "DD-MM-YYYY"
// Note: DIFFERENT from NAVAll.txt which uses "DD-MMM-YYYY" (e.g. "10-Jun-2026")
private static let historyDateFormatter: DateFormatter = {
    let f = DateFormatter()
    f.dateFormat = "dd-MM-yyyy"
    f.locale = Locale(identifier: "en_US_POSIX")
    f.timeZone = TimeZone(identifier: "Asia/Kolkata")  // NAV dates are IST
    return f
}()
```

### Pattern 3: SIP Accrual Engine — IST Date Math

**What:** Given a SIP with `dayOfMonth` and `lastAccruedDate`, enumerate all installment dates from `lastAccruedDate` (exclusive) up to today (IST, inclusive).

**IST Calendar for all date arithmetic:**

```swift
// [ASSUMED] Foundation Calendar — project-established pattern
var istCal = Calendar(identifier: .gregorian)
istCal.timeZone = TimeZone(identifier: "Asia/Kolkata")!
```

**Enumerate elapsed SIP dates:**

```swift
// [ASSUMED] Foundation Calendar
func elapsedInstallmentDates(
    dayOfMonth: Int,
    startDate: Date,       // SIP start (first possible accrual)
    lastAccruedDate: Date, // accrual cursor (exclusive lower bound)
    today: Date,           // IST start-of-day (inclusive upper bound)
    calendar: Calendar
) -> [Date] {
    var results: [Date] = []
    // Walk month-by-month from the month containing lastAccruedDate
    var cursor = calendar.startOfDay(for: lastAccruedDate)
    while true {
        // Move to next month if current cursor's day-of-month >= dayOfMonth already processed
        guard let candidate = calendar.date(
            bySetting: .day, value: dayOfMonth, of: cursor
        ) else { break }
        let clamped = candidate  // Calendar.date(bySetting:) clamps automatically (Feb 31 → Feb 28)
        if clamped <= lastAccruedDate { 
            // Advance cursor by one month and retry
            cursor = calendar.date(byAdding: .month, value: 1, to: cursor) ?? break
            continue
        }
        if clamped > today { break }
        if clamped >= startDate { results.append(clamped) }
        cursor = calendar.date(byAdding: .month, value: 1, to: cursor) ?? break
    }
    return results
}
```

**Day-of-month edge cases:**

- `Calendar.date(bySetting: .day, value: 31, of: someDate)` with `.gregorian` and IST timezone: if the target month has fewer than 31 days, Foundation clamps to the last day of that month (e.g., Feb 31 → Feb 28/29). [ASSUMED: Foundation behavior; consistent with Phase 3 NotificationScheduler which uses the same clamping approach for `UNCalendarNotificationTrigger`.]
- The `dayOfMonth` stored on SIP is the user's intent (e.g. "31st"). The actual transaction date in each month is clamped. Capture the actual date used in `Contribution.date` — important for the nearest-prior NAV lookup on months where clamping occurs.
- Weekend/holiday resolution: NPS and MF transactions submitted on weekends/holidays typically execute on the next working day in reality, but the exact execution date is not knowable without a broker feed. D-02's "nearest prior published NAV" fallback handles this: use the closest available NAV <= the SIP date. This is the stated reconcile-correctable estimate approach.

### Pattern 4: NPS Allocation Split and Decimal Rounding

**What:** Split a total SIP amount across 3 strategy holdings by allocation percentage.

**Algorithm:**

```swift
// [ASSUMED] Decimal arithmetic pattern
// allocationE + allocationC + allocationG must == 100 (validated on save)
// amounts are in paise (smallest unit) to avoid fractional-paisa issues
func splitAmount(_ total: Decimal, eP: Int, cP: Int, gP: Int) -> (e: Decimal, c: Decimal, g: Decimal) {
    // Compute shares as Decimal to avoid Double imprecision
    let hundred = Decimal(100)
    let rawE = total * Decimal(eP) / hundred
    let rawC = total * Decimal(cP) / hundred
    // Assign remainder to E (largest slice) to ensure sum == total
    let sumEC = roundDown2dp(rawE) + roundDown2dp(rawC)
    let gShare = total - sumEC  // residual, exact
    return (roundDown2dp(rawE), roundDown2dp(rawC), gShare)
}
// roundDown2dp: round to nearest paisa (2 decimal places), truncating, not banker's rounding
// NSDecimalNumberHandler with .down rounding mode
```

**Units calculation:**

```swift
// units = amount / NAV, with sufficient decimal places for NPS (4dp typical)
// [ASSUMED] NSDecimalNumber handler for controlled rounding
let handler = NSDecimalNumberHandler(
    roundingMode: .down,
    scale: 4,    // NPS NAV displayed to 4dp; units to 4dp is sufficient
    raiseOnExactness: false,
    raiseOnOverflow: false,
    raiseOnUnderflow: false,
    raiseOnDivideByZero: false  // guard nav != 0 before calling
)
let units = (amount as NSDecimalNumber)
    .dividing(by: nav as NSDecimalNumber, withBehavior: handler)
    .decimalValue
```

### Pattern 5: Monthly Reconcile Reminder

**What:** Schedule (or reschedule) a monthly `UNLocalNotification` after the first successful SIP accrual. Reuse `NotificationScheduler` + `ReminderInfo` directly.

**Existing API surface (confirmed by reading `NotificationScheduler.swift`):**

```swift

// ReminderInfo is a public struct with: id, title, date, isAllDay, recurrence, endRule, leadMinutes, noteID, blockID
// schedule(_:) is async throws
// cancel(reminderID:leadCount:weekdays:) is synchronous

let scheduler = NotificationScheduler(center: SystemNotificationCenter())

// Build a monthly "Reconcile SIP" reminder:
var info = ReminderInfo(
    id: sip.id,          // stable UUID — same UUID re-schedules (cancels old, adds new)
    title: "Confirm SIP units from statement",
    date: reconcileFireDate,  // e.g. SIP day + 3 days, at 09:00 IST, this month
    isAllDay: false,
    recurrence: ReminderRecurrence(type: .monthly),
    endRule: ReminderEndRule(type: .never)  // repeats forever until SIP isActive = false
)
// No noteID/blockID needed (this is an asset reminder, not a note reminder)
// The notification banner tap should deep-link to ReconcileView — requires a new

// Cancel when SIP.isActive set to false:

```

**Important:** `NotificationScheduler.makeRequest` always stamps `kReminderCategoryID` and routes taps to `kOpenNoteNotification`. For the reconcile reminder, the tap target is an Asset/SIP view, not a Note. Options:
1. Register a second `UNNotificationCategory` (e.g. `kReconcileCategoryID`) and handle in `NotificationActionDelegate`.
2. Reuse the existing deep-link channel but post a new `Notification.Name` for asset reconcile navigation.
3. Simplest: do not use `NotificationScheduler.makeRequest` internally — build the `UNNotificationRequest` directly in `SIPAccrualService`, using `UNUserNotificationCenter.current().add(request)` and store the identifier in `SIP.reminderNotificationID: String?`. This is self-contained within the SIP domain and avoids touching the Note notification machinery.

**Recommendation (Claude's discretion):** Option 3 — direct `UNUserNotificationCenter` in `SIPAccrualService`. The SIP reminder is a different domain (asset domain) from note reminders; adding a new category ID keeps concerns separated. Identifier scheme: `"sip-reconcile-\(sip.id.uuidString)"`.

```swift
    var isEstimate: Bool = true         // false after reconcile overwrites units
    var createdAt: Date = Date()
}
```

**Alternative NPS allocation storage:** Instead of 3 Int fields + 3 UUID fields on SIP (6 extra fields), a Codable Data? blob `npsAllocationData: Data? = nil` encoding `[(assetID: UUID, percent: Int)]` is more flexible but adds JSON encoding overhead. The 6-field flat approach is simpler and query-friendly; recommended given the fixed 3-strategy NPS structure.

### Anti-Patterns to Avoid

- **Double for NAV or money:** Always Decimal (project Pitfall 17). The npsnav.in API returns NAV as JSON Number — go through String intermediary `Decimal(string: String(format: "%.4f", doubleNAV))` or `NSDecimalNumber(value: doubleNAV).decimalValue`. Never `Decimal(exactly: doubleNAV)` — loses precision.
- **Stored enum in SchemaV8:** All type discriminators as `String` raw values (CloudKit rule 8).
- **`@Attribute(.unique)` anywhere:** CloudKit rule 2.
- **Single typealias flip:** STAB-08. All 10 typealiases (7 existing + 3 new) in one commit.
- **Skipping the 4 pbxproj edits:** New files silently absent from compile. Every new `.swift` needs 4 manual `project.pbxproj` entries.
- **Async in `didMigrate` closure:** `didMigrate` is synchronous (`throws`, not `async throws`). No `await` inside.
- **Accruing already-accrued dates:** `lastAccruedDate` must advance atomically with the `context.save()` on each batch of Contribution insertions. If save throws, rollback leaves `lastAccruedDate` unchanged — safe re-entry on next launch.
- **Blocking main thread:** Historical NAV fetches (potentially multiple URLSession calls) must run inside `Task {}` as with AMFINavService.

## Common Pitfalls

### Pitfall 1: npsnav.in NAV is JSON Number, Not String

**What goes wrong:** `JSONDecoder` decoding `nav` as `Decimal` or `String` fails. Using `Double` then casting to `Decimal` via `Decimal(exactly:)` loses fractional precision.
**Why it happens:** npsnav.in `/api/latest-min` uses bare JSON numbers (e.g. `49.5429`) while mfapi.in uses JSON strings (e.g. `"102.88080"`). Different sources, different types.
**How to avoid:** Decode `nav` as `Double`, then convert: `Decimal(string: String(format: "%.4f", doubleValue)) ?? Decimal(0)`. The 4dp format string preserves all significant digits present in NPS NAVs.
**Warning signs:** `nil` from `Decimal(string:)` on a value that looks valid, or slightly wrong NAVs on holdings.

### Pitfall 2: DD-MM-YYYY vs DD-MMM-YYYY Date Format Mismatch

**What goes wrong:** Both history endpoints (mfapi.in and npsnav.in) use `DD-MM-YYYY` (e.g. `"10-06-2026"`). NAVAll.txt uses `DD-MMM-YYYY` (e.g. `"10-Jun-2026"`). Reusing `AMFINavService.navDateFormatter` parses history dates silently as nil.
**Why it happens:** Different sources use different AMFI-derived date conventions.
**How to avoid:** Define a separate `historyDateFormatter` in the accrual service with `dateFormat = "dd-MM-yyyy"`. Never share formatters between services without checking the source format.
**Warning signs:** All history dates parse as nil → `navEntry(for:in:)` returns nil for every date → accrual writes `navUsed = 0` → units = 0 (infinite or zero).

### Pitfall 3: STAB-08 Typealias Partial Flip

**What goes wrong:** App crashes on save or query with `SwiftDataError` if any @Model typealias still points to SchemaV7 while the container uses SchemaV8.
**Why it happens:** The macro-expanded persistence code uses the typealias, not the SchemaVX.ModelName directly. Mixed typealiases → type mismatch at runtime.
**How to avoid:** grep for `typealias Asset =`, `typealias Note =`, etc. across all files in `Persistence/Models/` and confirm all 10 are on SchemaV8 before first build.
**Warning signs:** First launch after migration crashes at `ModelContainer` initialization, or saves throw `NSCocoaErrorDomain` errors.

### Pitfall 4: xcodeproj Explicit File Refs (4 Edits Per File)

**What goes wrong:** New `.swift` files compile in the IDE but are silently absent from the build target → missing symbols at link time, discovered only at post-merge build gate.
**Why it happens:** `MyHome.xcodeproj` has no synchronized folder groups; all file refs are explicit in `project.pbxproj`.
**How to avoid:** For each new `.swift` file, add: (1) PBXFileReference entry, (2) PBXBuildFile entry, (3) PBXSourcesBuildPhase ref in the app target, (4) PBXSourcesBuildPhase ref in the test target (if the file is referenced from tests via `@testable import`). Memory `[[xcodeproj-explicit-file-refs]]`.
**Warning signs:** `ld: symbol(s) not found for architecture arm64` or `Cannot find type 'SIP' in scope`.

### Pitfall 5: lastAccruedDate Advancing Before Save

**What goes wrong:** Accrual engine updates `sip.lastAccruedDate` optimistically before `context.save()`. If save throws, `lastAccruedDate` is already updated in-memory, so next accrual skips the failed dates.
**Why it happens:** `@Observable` object state is live; there is no automatic rollback of Swift value mutations.
**How to avoid:** Advance `lastAccruedDate` only after `try context.save()` succeeds. Or: collect all Contribution inserts for one date, attempt save, advance cursor on success. The cursor is the sole source of truth for "what's been accrued."
**Warning signs:** SIP installments silently missing from contribution log for a specific month after a network failure during accrual.

### Pitfall 6: Monthly UNCalendarNotificationTrigger Firing on Day > 28

**What goes wrong:** A repeating monthly trigger with `day = 31` never fires in months with fewer days (April, June, etc.) — iOS silently skips the trigger.
**Why it happens:** `UNCalendarNotificationTrigger(dateMatching:repeats: true)` does not clamp; it simply misses.
**How to avoid:** For the repeating reconcile reminder, clamp the notification day to `min(sipDayOfMonth + 3, NotificationScheduler.maxSafeMonthlyDay)` (which is 28). The +3 day offset gives statement time. For a SIP on the 31st, the reminder fires on the 28th of each month — acceptable. This is the same rationale as `NotificationScheduler.maxSafeMonthlyDay = 28` in the existing code.
**Warning signs:** Reconcile reminder not appearing in months that don't have enough days.

### Pitfall 7: npsnav.in Single-Maintainer Source Risk

**What goes wrong:** npsnav.in is maintained by one person (rishikeshsreehari); the site may go offline or the API may break without notice.
**Why it happens:** Single-maintainer open-source project, not an institutional source.
**How to avoid:** Same silent-fail + staleness-badge posture as AMFINavService (D-06/D-07). Never let a failed NPS fetch block UI or crash. The user always has manual NAV override available (existing `currentNAV` field in EditAssetView). NPSNavService returns without error on any network/parse failure, preserving the last cached NAV.
**Warning signs:** n/a — by design, no warning is surfaced; staleness badge signals age.

### Pitfall 8: mfapi.in Full History Response Size

**What goes wrong:** Fetching full NAV history for a fund active since 2008 returns 4,000+ records. If called for every MF fund per accrual session (e.g., multiple SIPs on different MF funds), this is a significant payload per session.
**Why it happens:** mfapi.in has no date-specific endpoint (verified); full history is the only option.
**How to avoid:** Cache the parsed history array in a session-scoped dictionary keyed by `amfiSchemeCode` within `SIPAccrualService`. One fetch per unique fund per accrual pass. Do not persist history to SwiftData — it's too large and stale tolerance is one session.
**Warning signs:** Accrual taking several seconds for users with many MF SIPs.

## Open Questions

1. **npsnav.in `latest-min` metadata shape**
   - What we know: the verified live response is a JSON array of `[code, nav]` pairs; the total entry in the fetch metadata mentions `lastUpdated: "10-06-2026"`. It is unclear whether this top-level key exists in the actual JSON root or is only visible in the `/api/latest` (full) endpoint.
   - What's unclear: whether `/api/latest-min` has a top-level `lastUpdated` or if it's a bare array.
   - Recommendation: Parse defensively — if the root is an array (bare), use today's IST date as the batch NAV date. If it's an object `{"data": [...], "lastUpdated": "..."}`, parse the date. Handle both in the parser.

2. **NPS Tier II scheme code pattern**
   - What we know: the user's NPS spans 3 strategies E/C/G under one PFM. The CONTEXT says "3 holdings." If the user has Tier II, codes differ.
   - What's unclear: whether the user has Tier I, Tier II, or both.
   - Recommendation: The NPS scheme picker should display both Tier I and Tier II variants (they appear in npsnav.in's scheme list with the naming convention in the scheme name). The user selects the right one. No code change needed beyond a good picker.

3. **SIPAccrualService re-entrancy if accrual takes > 1 scenePhase cycle**
   - What we know: AMFINavService uses an `isFetching: Bool` guard to prevent overlapping Tasks.
   - What's unclear: If accrual involves multiple sequential URLSession calls (one per MF fund history), the Task may still be running when the user backgrounds and re-foregrounds.
   - Recommendation: Add an `isAccruing: Bool` guard on SIPAccrualService, same pattern as AMFINavService's `isFetching`.

# Code Review: Phase 11.1 — SIP Automation & NPS NAV Auto-Refresh

Standard-depth review of the substantive new logic (accrual engine, NAV services,
reconcile loop, notification deep-link, SIP UI). Build is clean (0 errors) and the full
test suite passes (`-parallel-testing-enabled NO`). No Critical findings. Two Warnings
about accrual integrity under partial-failure, two Info notes.

## Warnings

### W-01 — A failed middle installment is permanently skipped
**File:** MyHomeApp/Support/SIPAccrualService.swift (`performAccrual`, the per-installment loop)

`lastProcessedDate` advances to the latest **successful** installment date even when an
earlier date in the same pass failed:
- MF branch uses `continue` on a NAV miss (line ~209/213) while still setting
  `lastProcessedDate` on later successes.
- NPS branch sets `lastProcessedDate` only when `allInserted`, but the outer date loop
  still continues to later dates.

Because the cursor (`sip.lastAccruedDate`) is then set to the latest success and the next
pass starts strictly after it, any failed date "in the middle" is never retried — the
installment is silently lost.

**Recommendation:** stop at the first failed installment (`break` out of the date loop)
so the cursor only advances over a *contiguous* successful prefix; the unresolved dates
retry on the next pass (once the NAV endpoint catches up).

**Severity rationale:** Warning, not Critical — the nearest-prior NAV fallback means a
miss is rare for recent installment dates in practice, but the failure mode is a silent
permanent gap in contributions/units.

### W-02 — NPS split is not atomic across the 3 strategy holdings
**File:** MyHomeApp/Support/SIPAccrualService.swift (NPS branch, lines ~157–197)

Within one installment, if strategy E succeeds (Contribution inserted + `stratAsset.units`
bumped) but C or G then fails (`allInserted = false; break`), E's mutations remain staged
in the `ModelContext`. The main context autosaves by default, so:
- If a later date succeeds, the partial E mutation is committed but C/G for that date are
  missing → the NPS holding's contributions drift out of step (E ahead by one installment).
- If **every** date is partial (e.g. C's scheme code is persistently unset), `lastProcessedDate`
  stays nil so the cursor never advances, yet autosave may still persist E's inserts →
  re-processing the same dates next pass **double-counts** E.

**Recommendation:** build the three (Contribution, unit-bump) pairs for an installment in
local temporaries and only `context.insert` / mutate `units` once all three strategies
resolve successfully; on any partial failure, apply nothing for that installment and stop.

## Info

### I-01 — Nearest-prior NAV lookup assumes newest-first ordering
**File:** SIPAccrualService.swift `navEntry(for:in:)` / `parseMFHistory` / `parseNPSHistory`

`navEntry` returns the first entry with `date <= target`, which is only the nearest-prior
NAV if the array is reverse-chronological. Parsers preserve endpoint order rather than
sorting. Correctness silently depends on mfapi.in / npsnav.in always returning newest-first.
Consider sorting `descending by date` after parse to be defensive.

### I-02 — `assertionFailure` on reconcile save failure
**File:** MyHomeApp/Features/Assets/ReconcileView.swift (`reconcile()`, ~line 122)

A `context.save()` failure triggers `assertionFailure` — a debug crash and a silent no-op
in release. Acceptable for a single-user household app, but a user-facing error toast would
be friendlier and is consistent with how other save paths could surface failure.

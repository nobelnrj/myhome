## NPS contribution split

| Option | Description | Selected |
|--------|-------------|----------|
| One SIP, auto-split by % | One monthly amount + allocation % per strategy; each installment splits and buys units in all 3 holdings at their own NAV. Mirrors the real single NPS debit. | ✓ |
| Three independent SIPs | Separate amount + date per strategy holding. More flexible, more to maintain, doesn't mirror the single deduction. | |

**User's choice:** One SIP, auto-split by %
**Notes:** MF SIPs are the degenerate case — one holding, 100% allocation. Allocation must sum to 100; rounding remainder handled deterministically by planner.

---

## Estimate NAV precision

| Option | Description | Selected |
|--------|-------------|----------|
| Exact NAV on the SIP date | Historical NAV lookup for the installment's date. npsnav.in has history; MF needs a historical endpoint (NAVAll.txt is latest-only). Closest to reality. | ✓ |
| Nearest NAV we already have | Latest cached NAV at app-open. No historical endpoint, single source, but drifts more before reconcile. | |

**User's choice:** Exact NAV on the SIP date
**Notes:** Deliberate, scoped extension of Phase 11's "single source / no second endpoint" stance — still zero new SPM deps. Fallback to nearest prior published NAV when exact date unavailable (holiday/weekend); log which NAV+date was used.

---

## Missed / paused SIPs

| Option | Description | Selected |
|--------|-------------|----------|
| Assume it ran; reconcile fixes it | Auto-accrue every scheduled month; statement reconcile corrects non-executed months. Plus isActive flag to stop a SIP. | ✓ |
| Skip + pause/stop controls | Per-installment skip + pause/resume/stop. More accurate hands-off, more UI to build. | |
| Confirm each installment | Prompt before each accrual. Most accurate, no longer hands-off. | |

**User's choice:** Assume it ran; reconcile fixes it (+ isActive stop flag)
**Notes:** No per-installment skip UI this phase; deferred.

---

## Reconcile mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| Total units of the holding | Type authoritative total units from statement; overwrites the running estimate. Log keeps estimate history. | ✓ |
| Per-month actual units | Correct each installment individually. Granular, but more typing and statements don't always break it out. | |

**User's choice:** Total units of the holding
**Notes:** Surfaced via monthly reminder through the Phase 3 Reminders hub → "confirm actual units" screen.

---

## SIP amount changes

| Option | Description | Selected |
|--------|-------------|----------|
| From the next installment only | Past months keep the amount in effect then (stored change history). New amount starts next cycle. | ✓ |
| Effective from a date I pick | Same model + backdating to a chosen effective date. More UI. | |
| Recompute all at new amount | Apply new amount to all past months. Simplest but wrong. | |

**User's choice:** From the next installment only
**Notes:** Amount-change history; each change effective from when made, so multiple changes stack correctly. Backdating deferred.

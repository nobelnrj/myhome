# Phase 11.1: SIP Automation & NPS NAV auto-refresh - Context

**Gathered:** 2026-06-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Two capabilities layered on the Phase 11 Asset Tracker (SchemaV7 `Asset` + `NetWorthSnapshot`):

1. **NPS NAV auto-refresh** — the one thing Phase 11 deliberately deferred (manual-only NPS, "npsnav single-maintainer source risk"). Add a daily best-effort NPS scheme-NAV refresh, mirroring `AMFINavService`: source from **npsnav.in** (open-source mirror of Protean/CRA — [rishikeshsreehari/npsnav](https://github.com/rishikeshsreehari/npsnav)), IST-gated daily fetch on `scenePhase .active`, silent-fail, cached value renders first, staleness badge signals age, manual override always available. `Asset` gains an `npsSchemeCode` (parallel to `amfiSchemeCode`); NPS holdings link via a searchable picker (mirror D-01/D-02). Each of the user's 3 NPS strategy holdings (E/C/G) links to its own scheme code.

2. **Hybrid SIP unit accrual (NPS + mutual funds)** — a SIP definition per holding, with an engine that **auto-accrues estimated units** on each elapsed SIP date and a **monthly reconcile reminder** (reuse the Phase 3 Reminders hub) that lets the user overwrite estimates with the real figure from their statement. Estimated contributions are first-class, logged, and clearly marked as estimates until reconciled.

**Schema:** one additive **SchemaV8** migration (SIP definition + amount-change history + contribution log + `Asset.npsSchemeCode`). Follow Phase 9/11 additive/CloudKit-ready rules; flip ALL `@Model` typealiases to V8 together (STAB-08).

**Out of scope (deferred / other phases):** stock auto-fetch (still manual); CloudKit/sharing (v2.0 trigger); any new SPM dependency (npsnav.in / mfapi.in are just URLs via URLSession — no package). Notes & Daily Routine (Phase 12) is untouched.

<decisions>
## Implementation Decisions

### NPS contribution modeling
- **D-01:** A monthly NPS contribution is modeled as **ONE SIP that auto-splits across the 3 strategy holdings by an allocation % the user sets** (e.g. E 50 / C 30 / G 20). Each installment splits the amount, then buys units in each strategy holding at *that holding's own NAV*. Mirrors the real single NPS debit-and-allocate. (NOT three independent per-strategy SIPs.)
  - Allocation %s must sum to 100; planner handles validation + the edge of a remainder paisa on split (assign rounding remainder deterministically, e.g. to the largest slice).
  - Mutual-fund SIPs are the simple case of this model: one holding, 100% allocation.

### Estimate NAV basis
- **D-02:** An auto-estimated installment uses the **exact NAV published on that installment's date** (historical lookup), not the latest cached NAV. Smallest gap before reconcile.
  - npsnav.in exposes per-scheme NAV history natively.
  - **MF historical NAV** requires a historical endpoint (AMFI historical NAV / mfapi.in) because the Phase 11 `NAVAll.txt` source is latest-only. **This is a deliberate, scoped extension of Phase 11's "single source, no second endpoint" stance** — still zero new SPM deps (another URL via URLSession). Planner picks the concrete historical endpoint + caching.
  - If the exact-date NAV is unavailable (holiday/weekend/not-yet-published), fall back to the nearest prior published NAV (the date the installment would actually transact). Capture which NAV + date was used in the contribution log.

### Missed / paused SIPs
- **D-03:** Auto-accrual **assumes every scheduled installment ran**; the monthly statement reconcile is the source of truth that corrects any month that didn't actually execute (paused/failed). No per-installment skip UI in this phase.
- **D-04:** A SIP carries an **`isActive` flag** — toggling it off stops future accrual entirely (the "I stopped this SIP" case). No separate pause/resume state machine.

### Reconciliation
- **D-05:** Reconcile is **whole-holding total-units overwrite**: the user types the authoritative total units (statements show this prominently) and it overwrites the running estimate for that holding. The contribution log retains the prior estimates as history. (NOT per-month installment-by-installment correction.)
- **D-06:** Reconcile is surfaced via a **monthly reminder through the existing Phase 3 Reminders hub**, opening a "confirm actual units from statement" screen. Reminder cadence is monthly; exact day relative to the SIP date is Claude's discretion (allow a few days for the statement to update).

### SIP amount changes
- **D-07:** Changing a SIP's amount applies **from the next installment only** — past months keep the amount that was in effect then. Amount changes are stored as **history** (each change effective from when it's made), so multiple changes over time stack correctly and past estimates stay faithful to what was really invested. (NOT recompute-all-at-new-amount.)
  - Backdating an amount change to a chosen effective date is NOT in scope this phase (user picked next-installment-only); note as a possible later refinement.

### Schema (SchemaV8, additive)
- **D-08:** One additive **SchemaV8** migration adds: `Asset.npsSchemeCode` (optional, parallel to `amfiSchemeCode`); a **SIP** model (per-Asset: amount, dayOfMonth, startDate, isActive, lastAccruedDate, plus NPS allocation %s where applicable); a **SIP amount-change history** record (effective-from + amount); a **Contribution** log (date, amount, navUsed, navDate, unitsAdded, isEstimate, link to Asset/SIP). Exact field shapes + how allocation %s are stored are Claude's discretion within these constraints. Follow FB13812722 `.custom`-over-`.lightweight`, additive-only, no `.unique` (CloudKit), and the STAB-08 typealias-flip-together rule.

### Claude's Discretion
- Exact SchemaV8 model field shapes (SIP, amount-change history, Contribution); how NPS allocation %s are persisted (on the SIP, or a small per-strategy split table).
- The concrete MF historical-NAV endpoint + caching strategy (D-02); the npsnav.in endpoint shape + parser.
- NPS scheme picker UX (search by PFM + asset class) — mirror the AMFI picker (D-01/D-02 of Phase 11).
- SIP "day-of-month" handling when the day exceeds month length (e.g. 31st in Feb) and weekend/holiday transaction-date resolution.
- Contribution-log UI (estimated vs reconciled timeline) placement and the reconcile screen layout.
- Reminder day-of-month relative to the SIP date.
- Rounding/remainder policy on the NPS allocation split and on units = amount ÷ NAV (Decimal).

</decisions>

<specifics>
## Specific Ideas

- NPS NAV source is **npsnav.in** (open-source mirror of Protean/CRA; CC BY-NC dataset — fine for this personal, non-commercial app). Third-party single-maintainer mirror → silent-fail + staleness posture is mandatory (same as AMFI).
- The user's NPS spans **3 strategies E/C/G under one PFM = 3 holdings = 3 scheme codes**, fed by one SIP split by allocation % (D-01).
- Estimates are **always labelled as estimates** until the user reconciles from the official statement; net worth uses the estimate in the meantime.
- Real-world drift confirmed this session: international/MF NAV-date timing differs across apps — exact-date NAV (D-02) + reconcile keeps our figure defensible.
- Zero new SPM deps; build/run iPhone 17 simulator, Xcode 26.5, scheme `MyHome` (memory `[[ios-build-simulator]]`). IST governs all "start of today" / installment-date / staleness comparisons.

</specifics>

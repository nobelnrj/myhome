<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → app gate | An unauthenticated person holding the device attempts to reach financial data |
| OS (LocalAuthentication) → app | LAContext returns success/error; the app must handle every error without dead-ending |
| app process → App Group UserDefaults | The lock-enabled preference crosses into shared defaults (a preference, not a secret — D5-07c) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Denial of Service | LockController.authenticate / mapError | mitigate | Every LAError case maps to a recoverable `LockAuthError?` (nil = stay-on-screen, never an inescapable loop); covered by LockStateTests cancel/lockout/failed/silent-fallback cases (SEC-02, D5-06) |
| T-05-02 | Denial of Service | LockController.authenticate (canEvaluate path) | mitigate | `.passcodeNotSet` from `canEvaluatePolicy` → `authError = .noPasscode` hard-block-with-escape; re-evaluated on next foreground (no data loss, no reinstall); covered by passcodeNotSetHardBlock test (D5-05, SEC-02) |
| T-05-03 | Elevation of Privilege | LockController init + scenePhaseChanged + enableLock/disableLock | mitigate | Cold launch with lockEnabled → isLocked=true; foreground >180s re-locks; enableLock/disableLock both require auth success before flipping the flag; covered by coldLaunchLocked/expiredGraceRelocks/enableLockAuthFailed tests (SEC-01, D5-01, D5-07a/b) |
</threat_model>

## Spacing Scale

iOS points (1pt = 1px on non-retina, 2px on @2x, 3px on @3x). All values are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4pt | Icon-to-label gaps, tight inline spacing |
| sm | 8pt | Compact element spacing inside rows |
| md | 16pt | Default horizontal padding; card internal padding (matches existing `.padding(16)` in SpendBudgetCard) |
| lg | 24pt | Section-level vertical rhythm |
| xl | 32pt | VStack spacing on unlock screen between icon group and button group |
| 2xl | 48pt | Not used in this phase |
| 3xl | 64pt | Not used in this phase |

Exceptions:
- **Minimum tap target:** 44pt × 44pt for all interactive elements (iOS HIG). Applied to the Unlock button (`.controlSize(.large)`), Settings List rows (system default), and chevron navigation buttons.
- **Unlock screen VStack spacing:** 32pt between major groups (icon+title cluster → error text → Unlock button). Source: RESEARCH.md UnlockView sketch.
- **Privacy blur radius:** 20pt (`.blur(radius: 20)`). Source: D5-02 / RESEARCH.md Pattern 3.

## Typography

All sizes are iOS Dynamic Type–compatible. Exact font roles map to SwiftUI font modifiers.

| Role | SwiftUI Modifier | Approx. Size | Weight | Line Height | Usage |
|------|-----------------|-------------|--------|-------------|-------|
| Display | `.largeTitle` | 34pt | Bold (`.bold()`) | 1.2 | Not used in Phase 5 |
| Heading | `.title2` | 22pt | Semibold (`.fontWeight(.semibold)`) | 1.2 | Unlock screen app name label |
| Body | `.body` | 17pt | Regular (400) | 1.5 | Settings List row labels, toggle label "Face ID Lock" |
| Supporting | `.subheadline` | 15pt | Regular (400) | 1.4 | Unlock screen error text, Settings section footers, About version string |
| Caption | `.caption` | 12pt | Regular (400) | 1.3 | No-passcode guidance text (D5-05 escape path) |

Rules:
- Never hand-roll font sizes. Always use semantic SwiftUI modifiers so Dynamic Type scales correctly.
- Heading weight is `.semibold` (600), body weight is regular (400). Exactly 2 weights used.
- Existing codebase uses `.font(.title2).fontWeight(.semibold)` for section headings (BudgetsView month label) — match this pattern in UnlockView app name.

## Color

This app uses iOS adaptive semantic colors exclusively. No hex values are hard-coded. All colors respond to light/dark mode automatically.

| Role | SwiftUI / UIKit Value | Proportion | Usage |
|------|----------------------|------------|-------|
| Dominant (60%) | `Color(.systemBackground)` | ~60% | Unlock screen full-screen background; NavigationStack background |
| Secondary (30%) | `Color(.secondarySystemBackground)` | ~30% | Settings List grouped background; existing card backgrounds (matches SpendBudgetCard pattern) |
| Accent (10%) | `.accentColor` (system blue) | ~10% | Unlock button fill (`.buttonStyle(.borderedProminent)`); active Toggle thumb; "Budgets" and "Manage Categories" row tint when presented as buttons |
| Destructive | `Color(.systemRed)` | Used only for destructive actions | Category delete confirmation inside ManageCategoriesView (existing, unchanged) |

Accent reserved for:
1. The "Unlock" button on UnlockView (`.borderedProminent` button style)
2. The Face ID Lock Toggle active state (system default)
3. "Manage Categories" and "Budgets" row tap-highlight tint in SettingsView

Semantic colors for error states on UnlockView:
- Error text (`.authenticationFailed`, `.biometryLockout`, `.noPasscode`, `.unknown`): `.foregroundStyle(.secondary)` — muted, not alarming
- No-passcode guidance text: `.foregroundStyle(.secondary)` at `.caption` size

## Surface Inventory

### Surface 1: UnlockView (new)

Full-screen overlay rendered when `lockController.isLocked == true && lockController.lockEnabled == true`.

**Layout (VStack, spacing: 32pt):**

```
[App icon — 80×80pt, RoundedRectangle cornerRadius 18]
[App name — "MyHome" — .title2 .semibold]
[Error text — .subheadline .secondary — visible only when authError != nil]
[Unlock button — .borderedProminent .controlSize(.large)]
[No-passcode guidance — .caption .secondary .multilineTextAlignment(.center) — visible only when authError == .noPasscode]
```

Background: `Color(.systemBackground)` filling `maxWidth: .infinity, maxHeight: .infinity`.

Transition: `.opacity` animation when appearing/disappearing.

App icon: use `Image("AppIcon")` clipped to `RoundedRectangle(cornerRadius: 18)`. This is the app's actual icon asset. Fall back to `Image(systemName: "house.circle.fill")` if the asset is unavailable.

**States:**

| State | What is shown |
|-------|--------------|
| Default (just appeared) | App icon + name + Unlock button. No error text. |
| Auth failed | + error message (see Copywriting Contract below) |
| Biometry locked out | + biometryLockout message |
| No passcode set | + noPasscode message + guidance text below Unlock button |
| Cancel / system cancel | No change; Unlock button stays visible and tappable |

**Behavior:**
- The system auth prompt is triggered automatically on `.active` scenePhase (Task { await lockController.authenticate() }) — banking-app feel.
- The visible Unlock button is always present as the manual retry path (D5-02 requirement). It is never hidden.
- On auth success, UnlockView disappears with `.opacity` transition.

### Surface 2: Privacy Blur Overlay (new, applied in RootView)

Not a separate view — a `.blur(radius: 20)` modifier applied to the TabView when `lockController.isBlurred == true`.

Activates on `.inactive` AND `.background` scenePhase (both, because iOS takes the app-switcher snapshot during `.inactive`). Deactivates on `.active`.

This is transparent to the user during normal use — they never see a "blurred" state; it only appears in the app switcher screenshot.

### Surface 3: SettingsView (new tab, tag 4)

SwiftUI `List` with `NavigationStack`. Presented as a tab item.

**Tab item:** `Label("Settings", systemImage: "gearshape")` at tag 4. Source: D5-12 discretion — `gearshape` is the iOS-standard settings icon used in the system Settings app, consistent with the existing SF Symbol tab style (house, list.bullet, chart.bar, note.text).

**List structure:**

```
NavigationStack {
    List {
        Section("Security") {
            Toggle("Face ID Lock", isOn: Binding(lockController.lockEnabled))
        }

        Section("Data") {
            Button("Manage Categories")  →  .sheet(ManageCategoriesView)
            Button("Budgets")            →  selectedTab = 2 (tab deep-link)
        }

        Section footer — "About" {
            Text("MyHome")
            Text("Version {version} ({build})")  ← Bundle.main
        }
    }
    .navigationTitle("Settings")
    .navigationBarTitleDisplayMode(.inline)
}
```

**Section breakdown:**

| Section | Rows | Interaction |
|---------|------|-------------|
| Security | "Face ID Lock" Toggle | isOn binding → auth-to-enable (D5-07a) / auth-to-disable (D5-07b) via Task |
| Data | "Manage Categories" Button row | Tap → .sheet presenting ManageCategoriesView |
| Data | "Budgets" Button row | Tap → selectedTab = 2 (dismisses SettingsView context, switches tab) |
| About (footer) | "MyHome" + version string | Non-interactive label; version from Bundle.main |

**Toggle wiring:** The toggle must NOT use a direct `isOn: $lockController.lockEnabled` binding. Instead, use a custom `Binding` that calls `Task { await lockController.enableLock() }` on true and `Task { await lockController.disableLock() }` on false. This is required because auth is async (D5-07a/b). The toggle visually reflects `lockController.lockEnabled` (the underlying @Observable property) and snaps to the persisted value after auth completes.

**Category entry presentation:** Sheet (not NavigationLink push). Source: D5-12 discretion — sheet matches the existing BudgetsView presentation of ManageCategoriesView; keeps consistent UX across both callers.

**Budgets row appearance:** Plain `Button` styled as a List row with a disclosure chevron. Use `HStack { Text("Budgets"); Spacer(); Image(systemName: "chevron.right").foregroundStyle(.tertiary) }` inside a `Button` or a Label row — not a NavigationLink (NavigationLink would push a child view; this is a tab switch, not a push).

## Interaction Contract

### Unlock Flow

1. Cold launch with lock enabled → LockController.init sets `isLocked = true`.
2. RootView `.onChange(of: scenePhase)` fires `.active` → auto-triggers `Task { await lockController.authenticate() }`.
3. System Face ID / passcode prompt appears.
4. On success → `isLocked = false` → UnlockView disappears via `.opacity` transition.
5. On failure / cancel → `authError` set → UnlockView stays; error text updates; Unlock button remains tappable.
6. User taps Unlock button → same `authenticate()` cycle repeats.

### Re-lock on Background

1. App goes to `.inactive` → `isBlurred = true`; `backgroundedAt` stamped.
2. App goes to `.background` → `isBlurred = true` (already true; idempotent).
3. App returns to `.active` with elapsed time > 180s → `isLocked = true`.
4. App returns to `.active` with elapsed time ≤ 180s → no re-lock; `isBlurred = false`.

### Toggle Enable

1. User flips toggle ON in SettingsView.
2. Custom Binding setter calls `Task { await lockController.enableLock() }`.
3. System auth prompt appears: "Verify your identity to enable the lock."
4. On success → `lockController.lockEnabled = true` (UserDefaults write) → toggle stays ON.
5. On failure → `lockController.lockEnabled` stays false → toggle reverts to OFF.

### Toggle Disable

1. User flips toggle OFF in SettingsView.
2. Custom Binding setter calls `Task { await lockController.disableLock() }`.
3. System auth prompt appears: "Verify your identity to disable the lock."
4. On success → `lockController.lockEnabled = false` (UserDefaults write) → toggle stays OFF.
5. On failure → `lockController.lockEnabled` stays true → toggle reverts to ON.

### Category Management

1. User taps "Manage Categories" row in SettingsView.
2. `showManageCategories = true` → `.sheet(isPresented:) { ManageCategoriesView() }` presents.
3. ManageCategoriesView is fully self-contained (has its own NavigationStack + dismiss button).
4. Sheet dismisses on Done / swipe-down.

### Budgets Deep-Link

1. User taps "Budgets" row in SettingsView.
2. `selectedTab = 2` → TabView switches to BudgetsView.
3. No sheet, no push — direct tab switch. SettingsView is hidden (not dismissed).

## Accessibility Contract

- All interactive elements have system-provided accessibility labels via SwiftUI semantics.
- UnlockView Unlock button: accessible label "Unlock" (button title = label).
- App icon on UnlockView: `.accessibilityHidden(true)` — decorative; screen reader does not need to read it.
- App name label on UnlockView: `.accessibilityAddTraits(.isHeader)`.
- Error text on UnlockView: announced dynamically when `authError` changes via `.accessibilityAnnouncement` or by ensuring the Text view is part of the view hierarchy (SwiftUI will announce changes via VoiceOver focus).
- Settings List rows: inherit system accessibility behavior (List row accessibility, toggle accessibility) — no overrides needed.
- Face ID Lock Toggle: VoiceOver reads "Face ID Lock, switch, off/on" — system behavior; no custom label needed.
- Minimum tap target 44pt satisfied by `.controlSize(.large)` on Unlock button, by List row intrinsic height, and by existing chevron button frames.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → app open | Unauthenticated holder of the device attempts to reach financial data on launch/foreground |
| app → app-switcher (OS) | iOS captures a snapshot during `.inactive`; content must not leak |
| SettingsView → LockController | The lock-enable/disable control must not flip the flag without auth |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Denial of Service | UnlockView (always-visible Unlock button) | mitigate | UnlockView always renders a tappable Unlock button and mapped error text — the user can always re-trigger auth; never a blank/auto-only gate (D5-02, D5-06). Verified by build + manual UAT item 2/7 |
| T-05-02 | Information Disclosure / DoS | UnlockView no-passcode guidance | mitigate | When `authError == .noPasscode`, UnlockView shows reachable guidance to set a device passcode and re-evaluates on next foreground — escapable, no data loss (D5-05). UAT item 7 |
| T-05-03 | Tampering | SettingsView toggle + RootView lockController ownership | mitigate | Toggle uses a custom Binding that calls auth-gated `enableLock()`/`disableLock()` — flag never flips without auth success; persistence round-trips through App Group UserDefaults; covered by LockSettingsTests (enableLockBlockedOnAuthFailure, disableLockClearsFlagOnAuthSuccess, lockEnabledPersistsToDefaults) (D5-07a/b/c) |
| T-05-04 | Information Disclosure | RootView privacy blur overlay | mitigate | `.blur(radius: 20)` keyed on `lockController.isBlurred`, set true on BOTH `.inactive` and `.background` so the app-switcher snapshot (taken during `.inactive`) is obscured (D5-02, Pitfall 1). Verified by manual UAT item 3 (OS-level, not unit-testable) |
</threat_model>

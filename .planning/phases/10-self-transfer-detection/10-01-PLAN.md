## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scorer → persisted pending pairs | Scorer output mutates financial records (transferPairID); a wrong pair routes real spend toward exclusion |
| existing AccountBalance formula → confirmed-transfer balance-move | If the sign convention is wrong, a confirmed pair corrupts displayed net worth |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | TransferDetectionScorer false-positive pairing | mitigate | Hard AND-rules (D-01..D-05) each independently unit-tested; scorer only writes a PENDING link (isTransfer stays nil) — nothing is excluded until user confirms (XFER-02 gate, Plan 03) |
| T-10-02 | Tampering | Re-surfacing confirmed/rejected legs on re-scan | mitigate | STAB-08-safe Swift `.filter { $0.isTransfer == nil }` (NO #Predicate on Bool?); secondScanIdempotent + confirmedLegSkipped + rejectedLegSkipped tests prove non-nil legs are never re-evaluated |
| T-10-03 | Elevation | Double-counted / sign-wrong balance-move corrupting net worth | mitigate | netWorthUnchangedByTransfer test asserts the pair nets to zero; Task 4 blocking human-verify validates the savings sign convention against live data BEFORE Plan 03 wires D-16 |
| T-10-04 | Tampering | Half-written pair on save failure (dangling transferPairID) | mitigate | Both legs mutated BEFORE a single `try context.save()` (CR-01); SwiftData save is atomic per call so both writes commit or roll back together |
</threat_model>

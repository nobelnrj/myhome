<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pending pair → user confirm gate | The confirm/reject swipe is the ONLY thing that excludes a transfer from spend; a mis-rendered or un-actionable row would either hide real spend or trap a pair |
| sync completion → scorer invocation | An incorrectly wired hook could skip detection or run it across an await boundary (STAB-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-08 | Tampering | A confirmed pair becomes un-hideable / stays in default spend list | mitigate | filteredExpenses `.normal` excludes isTransfer == true; Task 4 human-verify confirms a confirmed pair leaves the default list and appears only under Transfers |
| T-10-09 | Repudiation | Silent exclusion (a pair hidden without user action) | mitigate | Scorer only writes a PENDING link; nothing is isTransfer == true until the user swipe-confirms (D-11 "nothing silently excluded"); pending pairs always surface in the inbox section |
| T-10-10 | Tampering | Half-resolved pair (one leg confirmed, other dangling) | mitigate | confirmPair/rejectPair mutate BOTH legs before one atomic save (CR-01); defensive in-memory credit-leg resolution skips rendering a row whose counterpart vanished |
| T-10-11 | Tampering | Scorer skipped after sync or run across await (STAB-02) | mitigate | scan() called after all await boundaries and the final save; synchronous @MainActor call — no STAB-02 exposure |
</threat_model>

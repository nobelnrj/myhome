<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app → OS notification center | Scheduled/cancelled requests + registered category cross into UNUserNotificationCenter (64-pending cap) |
| OS notification → app (deep-link) | Tap/action payload routes back into the app; identifiers must map to a valid note/row |
| user integers (lead-time, after-N, weekdays) → scheduler | User-controlled inputs bounded before scheduling |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-13 | Tampering / DoS (64-cap) | recurrence + lead-time + after-N scheduling | mitigate | Clamp lead-minutes ≥ 0, after-N ≥ 1, weekday set within 1–7; rely on 03-03's 64-cap budget; reschedule = cancel-then-rebuild to avoid orphan accumulation |
| T-03-14 | Spoofing / wrong-target | deep-link / Complete routing | mitigate | Deterministic reminder identifiers map to a specific note/row id; Complete checks only the identified row |
| T-03-15 | Information Disclosure | sensitive note text on lock-screen notification | accept | Acceptable for single-device household v1 (RESEARCH Security Domain V13); flag lock-screen redaction as a v2 consideration |
| T-03-16 | Information Disclosure | note/PII content in logs | mitigate | No logging of note title/body; notification content not echoed to console (zero-telemetry charter) |
| T-03-SC | Tampering | dependency installs | accept | No packages added — first-party UserNotifications/SwiftUI only |

# Requirements (DRAFT) — Milestone v1.4: Finance & AI Depth

> **DRAFT — not yet active.** This is a forward-looking plan written while v1.3.1 (Phases 23-24) is
> still executing. It will be reviewed and finalized via `/gsd-new-milestone` once v1.3.1 ships —
> REQ-IDs, phase numbers, and scope here may still change. Do not treat as authoritative until
> promoted to `.planning/REQUIREMENTS.md`.

**Milestone goal:** Pay down two shipping-feature security gaps first, then deepen the app's finance
capability (recurring detection, bill reminders, exports, a global account scope) and make the
on-device AI insight genuinely smarter and better-grounded in the user's real data.

**Constraints carried from v1.0–v1.3.1:** local-only (still no cloud/CloudKit — that remains the
paid trigger), zero recurring cost, no paid third-party services/SDKs, no analytics/telemetry, free
public data only with manual override always available, CloudKit-ready SwiftData schema (additive,
no `.unique`, optional/defaulted), iOS 17 deployment floor, zero new paid dependencies. On-device AI
stays Apple FoundationModels only — no external LLM calls, no new paid AI dependency.

**v1.4-specific ground rules:**
- **Security first is a hard ordering constraint.** #31 and #43 are known, already-shipped defects
  (one un-merged fix, one live trust gap in auto-sync). They land before any feature work in this
  milestone, no exceptions.
- **Schema bump is allowed this milestone** (unlike v1.3.1). Current schema is SchemaV11; recurring-
  transaction detection is expected to require SchemaV12. Every other item is expected to avoid a
  schema bump — flagged per requirement/phase below.
- **Exports are on-device only.** No upload, no cloud storage integration — the app hands the user a
  file via the system share sheet; where it goes after that is up to them.
- **Smarter AI stays grounded.** Any AI-authored insight/narrative must be verifiable against the
  user's real SwiftData-backed figures (continuing the Phase 16 numeric-integrity verifier pattern) —
  no model-invented numbers, no external network calls.

## v1.4 Requirements

### Security Debt (SEC)

<!-- Two known, already-diagnosed defects on shipped features. Land FIRST — hard ordering constraint. -->

- [ ] **SEC-01**: The four un-merged Phase-05 Face ID code-review fixes (CR-01, CR-02, CR-03, WR-02
  from `LockController.swift`, and WR-01 from `SettingsView.swift`) are merged onto `main`: the App
  Group `UserDefaults` suite is resolved once at `init` (not re-resolved per access), all
  `canEvaluate` failure paths surface `authError` (not just `.passcodeNotSet`), `enableLock()` /
  `disableLock()` apply the same `canEvaluate` preflight as `authenticate()`, `markBackgrounded(at:)`
  compiles only in `#if DEBUG`, and the Face ID toggle in Settings guards against re-entrant taps
  while an auth call is in flight. (#31)
- [ ] **SEC-02**: Auto-sync no longer trusts any peer advertising the same Multipeer service on the
  LAN. A paired-devices allowlist is established via an explicit first-time pairing exchange between
  the two household phones; peer identity persists across app launches; snapshots/connections from an
  unpaired peer are rejected or ignored; a pairing UI exists in Settings › Sync. (#43)

### Recurring Transactions & Bill Reminders (REC, BILL)

<!-- Grouped: a detected recurring series is the natural source for a bill reminder. -->

- [ ] **REC-01**: The app automatically detects recurring transactions (subscriptions, EMIs, SIPs)
  from the combined ingested + manual expense stream, grouping matching transactions (same/similar
  merchant, similar amount, regular interval) into a recurring series. (#28)
- [ ] **REC-02**: Detected recurring series are surfaced in the UI — flagged on the expense row and
  visible as a grouped list — with a way to confirm, correct, or dismiss a detected grouping (mirrors
  the existing self-transfer confirm-not-silent pattern from v1.1). (#28)
- [ ] **BILL-01**: User can set up a bill/subscription reminder, optionally seeded from a REC-01
  detected recurring series (amount + expected next date pre-filled) or created manually for a bill
  the app cannot yet detect (e.g. paid outside the ingested email stream). (#27)
- [ ] **BILL-02**: Bill reminders fire as local notifications ahead of the expected due date (reusing
  the Phase 3 local-notification infrastructure), with a management view to see, edit, snooze, or
  cancel upcoming reminders. (#27)

### Data Exports (EXP)

<!-- On-device only — share sheet hand-off, no cloud upload. -->

- [ ] **EXP-01**: User can export expense data (date range selectable) to **CSV**, formatted for
  spreadsheet import (INR amounts as plain decimals, ISO or locale dates, category/tag/account
  columns). (#29)
- [ ] **EXP-02**: User can export the same data as **JSON** (a structured, complete dump — expenses,
  and optionally accounts/assets) for a technical user or a future re-import path. (#29)
- [ ] **EXP-03**: Export is handed off via the system share sheet (AirDrop, Files, Mail, etc.) —
  generated on-device, no server round-trip, no analytics on what's exported. (#29)

### Global Account Scope (SCOPE)

<!-- Extends OVF-01..03 (v1.3) app-wide. UI/state layer only — no schema bump expected. -->

- [ ] **SCOPE-01**: A single shared `AccountScope` observable (injected via `.environment`) replaces
  Overview's local `filter.accountIDs` and Expenses' local `accountFilter` as the one source of truth
  for the account dimension; selecting an account in one screen is reflected in the other. (#44)
- [ ] **SCOPE-02**: Expenses surfaces the account scope as a visible header pill (the same
  `OverviewScopePill` component, account-only mode) placed between the filter funnel and the `+` add
  icon; the funnel keeps category/transfer filtering. Overview's custom date range remains
  Overview-local (not part of the shared scope, since Expenses is already date-grouped). (#44)

### Smarter AI (AI)

<!-- Continues v1.2's AI-01..05 numbering. Stays on-device (FoundationModels), grounded in real data. -->

- [ ] **AI-06**: The on-device AI insight (currently Analytics-only, Phase 16) is also surfaced on the
  Overview dashboard, using the same availability gating, streaming, and numeric-integrity
  verification already proven in Phase 16. (#30)
- [ ] **AI-07**: The AI insight supports follow-up questions — the user can ask a bounded follow-up
  and get a grounded, on-device answer derived from the same verified figures, not free-form model
  invention. (#30)
- [ ] **AI-08**: The AI insight can generate a multi-month trend narrative (e.g. "spend on X has risen
  for 3 months"), computed from real aggregated history and passed through the numeric-integrity
  verifier before display. (#30)
- [ ] **AI-09**: The existing insight-generation logic is fixed so it no longer miscounts investment
  contributions (SIPs, asset purchases) as spend — the underlying aggregation must exclude
  investment-classified transactions the same way it already excludes confirmed self-transfers. (#30)

## Out of Scope (explicit exclusions)

- **CloudKit / iCloud sync / cloud-hosted exports** — cloud remains the future paid-upgrade trigger;
  v1.4 exports are local file hand-off only.
- **Cross-household pairing / N-device allowlist** — SEC-02's allowlist is scoped to exactly the two
  household phones, not a general multi-device pairing system.
- **Automatic bill payment / bank-initiated actions** — reminders only; the app never initiates a
  payment.
- **External LLM calls for AI insight (#30)** — on-device FoundationModels only, per the established
  free/private/no-new-paid-dependency constraint.
- **PDF export** — deferred; CSV + JSON are the v1.4 formats. PDF (styled report) can be considered
  in a later milestone if requested.
- **Recurring-detection auto-creating reminders without confirmation** — mirrors the v1.1
  self-transfer precedent: auto-detect, always confirm, never silent.
- **Barcode/receipt scanning, multi-currency, Android** — unchanged standing exclusions from
  `PROJECT.md`.

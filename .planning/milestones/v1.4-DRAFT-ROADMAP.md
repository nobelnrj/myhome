# Roadmap (DRAFT) — Milestone v1.4: Finance & AI Depth

> **DRAFT — not yet active.** Written while v1.3.1 (Phases 23-24) is still executing, so nothing here
> disturbs the active milestone's `ROADMAP.md`/`REQUIREMENTS.md`. This will be reviewed, re-sequenced
> if needed, and formally adopted via `/gsd-new-milestone` once v1.3.1 ships. Phase numbers assume
> v1.3.1 finishes at Phase 24, so v1.4 starts at **Phase 25**.

## Overview

v1.4 pays down two known security gaps on already-shipped features before doing any new feature
work — four un-merged Face ID review fixes (#31) and a real incident where auto-sync's "trust any
LAN peer" model let a seeded simulator pollute a phone's Kitchen data (#43). With that debt cleared,
the milestone deepens finance: recurring-transaction detection feeding bill/subscription reminders,
CSV/JSON data export, and promoting the account-scope concept from Overview-only (v1.3 OVF-01..03)
to a shared app-wide state also visible in Expenses. It closes with smarter on-device AI — surfacing
the existing Phase 16 insight on the Overview itself, adding grounded follow-up questions and
multi-month trend narratives, and fixing a real bug where the insight miscounts investment
contributions as spend. Everything stays local-first and free: on-device FoundationModels only for
AI, no cloud, no new paid dependencies. Schema bump (SchemaV11 → V12) is expected for the recurring-
transaction work — the first schema change allowed since v1.3's SchemaV11, and the first since
v1.3.1 (which deliberately made none).

## Phase Details

### Phase 25: Security Hardening — Face ID Fixes & Sync Allowlist
**Goal**: The two known, already-diagnosed security gaps on shipped features — dormant Face ID
review fixes and auto-sync's any-LAN-peer trust model — are closed, with no regression to existing
lock or sync behavior.
**Depends on**: Phase 24 (v1.3.1 complete) — first phase of v1.4. No functional dependency on later
v1.4 phases; this is sequenced first purely because security debt is a hard ordering constraint for
this milestone.
**Requirements**: SEC-01, SEC-02
**Schema impact**: None expected. SEC-02's paired-device identity is expected to be stored outside
SwiftData (Keychain/UserDefaults, mirroring how lock state and Gmail tokens are already stored) —
explicitly avoiding a schema bump for a security-scoped phase. If implementation instead needs a
SwiftData-tracked pairing record, that would require a schema bump and should be flagged during
planning.
**Success Criteria** (what must be TRUE):
  1. `LockController`'s App Group `UserDefaults` suite is resolved once at `init`, not re-resolved on
     every read/write, and no lock-state read/write can silently fall back to `.standard` mid-session.
  2. Every `canEvaluate` failure path (not just `.passcodeNotSet`) surfaces a user-visible
     `authError`, including during `enableLock()`/`disableLock()`, which apply the same preflight as
     `authenticate()`.
  3. `markBackgrounded(at:)` no longer compiles into release builds, and the Face ID toggle in
     Settings cannot be driven re-entrantly while an enable/disable auth call is in flight.
  4. Only the two paired household phones connect and merge over auto-sync — a third device (e.g. a
     freshly seeded simulator or an unpaired phone) advertising the same Multipeer service is
     rejected or ignored, verified by a repeat of the #43 incident scenario.
  5. Pairing is established through an explicit, visible action in Settings › Sync (not implicit
     first-contact trust), and paired identity survives an app relaunch.

### Phase 26: Recurring Detection & Bill Reminders
**Goal**: The app recognizes recurring transactions (subscriptions, EMIs, SIPs) in the expense
stream and turns a recurring bill — detected or manual — into a scheduled local-notification
reminder, following the confirm-not-silent pattern already proven for self-transfer detection.
**Depends on**: Phase 25 (security lands first per milestone ordering); functionally independent of
the security fixes. Reuses Phase 3's local-notification infrastructure and Phase 7's ingestion
pipeline/merchant normalization as detection input.
**Requirements**: REC-01, REC-02, BILL-01, BILL-02
**Schema impact**: **SchemaV12 expected.** Needs a way to group transactions into a recurring series
(e.g. a `RecurringSeries` model or a `recurringGroupID` field on `Expense`) and a `BillReminder`
model (amount, expected next date, notification identifier, source series link). This is the first
schema bump since SchemaV11 (v1.3) — additive/optional fields only, CloudKit-ready, per the
project's durability constraint.
**Success Criteria** (what must be TRUE):
  1. A recurring pattern (same/similar merchant + similar amount + regular interval) across
     historical expenses is grouped into a detected recurring series, surfaced as such in the UI.
  2. A detected grouping can be confirmed, corrected, or dismissed by the user — nothing is silently
     auto-applied (mirrors v1.1's self-transfer UX precedent).
  3. User can create a bill reminder seeded from a confirmed recurring series (amount/date
     pre-filled) or fully manually for a bill outside the ingested email stream.
  4. Bill reminders fire as local notifications ahead of the expected due date, and a management view
     lets the user see, edit, snooze, or cancel any upcoming reminder.
  5. Recurring-series detection and reminder scheduling both run entirely on-device with no new
     network calls.

### Phase 27: Data Export
**Goal**: The user can get their expense data out of the app in a spreadsheet-friendly (CSV) and a
complete structured (JSON) format, handed off entirely on-device via the system share sheet.
**Depends on**: Nothing new functionally — reads existing Expense/Category/Account/Asset models
directly. Sequenced after Phase 26 per this milestone's declared phase order (finance-depth features
grouped before the scope-pill/AI phases).
**Requirements**: EXP-01, EXP-02, EXP-03
**Schema impact**: None — export is read-only against the existing schema; no new persisted state.
**Success Criteria** (what must be TRUE):
  1. User can export expenses for a selected date range to a CSV file with INR amounts as plain
     decimals and category/tag/account columns, importable into a spreadsheet app without manual
     cleanup.
  2. User can export the same (or broader — expenses + accounts/assets) data as a structured JSON
     file.
  3. Both exports are handed off via the system share sheet (AirDrop/Files/Mail/etc.); no server
     round-trip and no analytics on export content or frequency.
  4. Export completes correctly for an empty range (no data) without crashing, and for a large
     date range (all-time) without unacceptable delay.

### Phase 28: Global Account Scope
**Goal**: Selecting an account is a single, shared app state — pick "HDFC" once and both Overview and
Expenses reflect it — with Expenses surfacing the same scope pill component Overview already has.
**Depends on**: Phase 21 (v1.3 `OverviewFilter`/`AccountFilter` engine and `OverviewScopePill`
component, which this phase generalizes to an app-wide `@Observable`). Sequenced after Phase 27.
**Requirements**: SCOPE-01, SCOPE-02
**Schema impact**: None — this is a view/state-layer change (a shared `@Observable` injected via
`.environment`), not a persistence change. (Last-selected account may be cached in `UserDefaults` for
continuity across launches, which is not a schema concern.)
**Success Criteria** (what must be TRUE):
  1. Selecting an account (or subset) on Overview is immediately reflected in Expenses' scope, and
     vice versa — one shared `AccountScope` source of truth, no divergent local state left behind.
  2. Expenses shows the account scope as a header pill (same component as Overview's), positioned
     between the filter funnel and the `+` add icon; the funnel still handles category/transfer
     filtering.
  3. Overview's custom date range remains Overview-local and is unaffected by the shared account
     scope becoming app-wide.
  4. All-accounts remains the default scope on fresh launch, matching the existing OVF-01 default.

### Phase 29: Smarter AI — Overview Insight, Follow-Ups & Trends
**Goal**: The on-device AI insight moves beyond an Analytics-only card into a genuinely more useful
assistant — visible on the Overview, answering bounded follow-up questions, narrating multi-month
trends, and no longer miscounting investment contributions as spend — all grounded in real figures
and verified on-device.
**Depends on**: Phase 16 (v1.2 FoundationModels stack: availability gating, `@Generable` guided
generation, numeric-integrity verifier, streaming) which this phase extends. Also depends on Phase 26
(recurring-series data available as grounding input for trend/bill-aware narratives) and Phase 21
(Overview filter context the insight should respect when generating a scoped insight). Sequenced
last in this milestone.
**Requirements**: AI-06, AI-07, AI-08, AI-09
**Schema impact**: None expected — reads existing Expense/Account/Asset/RecurringSeries data; any
insight cache is expected to live outside SwiftData (in-memory or `UserDefaults`), not as a new model.
**Success Criteria** (what must be TRUE):
  1. The same AI insight card (with its existing availability gating and streaming) is surfaced on
     the Overview, not just Analytics, reflecting whatever account/date scope is currently active.
  2. The user can ask a bounded follow-up question and receive an on-device, grounded answer — no
     free-form model invention of figures not present in the user's real data.
  3. The insight can narrate a real multi-month trend (e.g. rising spend in a category across 3+
     months), and every number in that narrative passes the existing numeric-integrity verifier
     before display.
  4. The insight no longer counts investment-classified transactions (SIPs, asset purchases) as
     spend — verified against a case that previously miscounted it.
  5. No insight generation makes an external network call; on unavailable devices, the existing
     silent-absence/templated-fallback behavior from Phase 16 is preserved.
